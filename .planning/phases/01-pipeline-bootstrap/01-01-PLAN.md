# Plan 01-01 — Bootstrap webatlas-pipeline on HPC

## Objective

Install and run haniffalab/webatlas-pipeline (Nextflow) on HPC against its own
example data, pin all versions/containers, and write down exactly what AnnData
and params-YAML the pipeline expects and what Vitessce output it emits — so
Phase 2 builds inputs against a known contract instead of guessing.

## Context

- Pipeline: https://github.com/haniffalab/webatlas-pipeline (Nextflow → Vitessce).
- HPC uses SLURM + Singularity (no root Docker). Nextflow must use the
  `singularity` profile / container engine.
- Scratch for large intermediates: `/vast/scratch/users/kriel.j/`.

## Tasks

<task id="1" type="execute">
<title>Clone pipeline and pin versions</title>
<read_first>
- venture_atlas/PROJECT.md
- CLAUDE.md
</read_first>
<action>
Clone the pipeline into `venture_atlas/webatlas/webatlas-pipeline` at a specific
tagged release (not a moving `main`). Record in
`venture_atlas/webatlas/versions.env`:
  WEBATLAS_TAG=<chosen release tag>
  WEBATLAS_COMMIT=<git rev-parse HEAD>
  NEXTFLOW_VERSION=<nextflow -version>
  CONTAINER_ENGINE=singularity
Load or install Nextflow (Java 17+). Confirm `nextflow -version` prints.
</action>
<acceptance_criteria>
- `venture_atlas/webatlas/webatlas-pipeline/main.nf` exists.
- `venture_atlas/webatlas/versions.env` contains WEBATLAS_COMMIT= and NEXTFLOW_VERSION= with non-empty values.
- `git -C venture_atlas/webatlas/webatlas-pipeline describe --tags` returns the pinned tag.
</acceptance_criteria>
</task>

<task id="2" type="execute">
<title>Run the pipeline example under Singularity/SLURM</title>
<read_first>
- venture_atlas/webatlas/webatlas-pipeline/main.nf
- venture_atlas/webatlas/webatlas-pipeline/nextflow.config
- any example params under venture_atlas/webatlas/webatlas-pipeline (docs/tests/templates)
</read_first>
<action>
Write `venture_atlas/webatlas/run_example.sh` — an sbatch script that runs the
pipeline's bundled example (or the smallest documented example params file) with
`-entry Full_pipeline`, `-profile singularity` (or the repo's HPC profile), and
`--outdir /vast/scratch/users/kriel.j/webatlas_example_out`. Submit it. Capture
the SLURM job id and logs into `venture_atlas/webatlas/logs/`.
</action>
<acceptance_criteria>
- `run_example.sh` contains `nextflow run`, `-entry Full_pipeline`, and a singularity profile flag.
- The job completes: `--outdir` contains a Vitessce config JSON and at least one `.zarr`/`.ome.zarr` output.
- Nextflow reports `Succeeded` for all processes (grep the log for a completion/summary line, no `FAILED`).
</acceptance_criteria>
</task>

<task id="4" type="execute">
<title>Document the pipeline I/O contract</title>
<read_first>
- the example params YAML used in task 2
- the generated Vitessce config JSON
- pipeline docs for AnnData / h5ad expectations
</read_first>
<action>
Write `venture_atlas/webatlas/PIPELINE_CONTRACT.md` capturing, with concrete
field names from the actual run:
  - The params-YAML schema: outdir, projects→datasets→data-type→data-path
    nesting, and the `vitessce_options` keys (spatial coords mapping, obsSets,
    feature/matrix and embedding options, layout template).
  - The AnnData conventions the h5ad ingestor expects: where it reads spatial
    coords (`obsm` key), how it reads categorical obs as obsSets, how it reads
    the expression matrix / `var`, and whether/how multiple feature matrices or
    a feature-type are supported (this answers Risk #2 for metabolites).
  - Which zarr format the output uses (v2 vs v3) — note if down-conversion is needed.
</action>
<acceptance_criteria>
- `PIPELINE_CONTRACT.md` contains a "params YAML" section with the real keys from run_example.sh's params file.
- It contains an "AnnData expectations" section naming the exact `obsm` spatial key and obsSets mechanism.
- It contains an explicit statement on multi-feature-matrix / feature-type support (yes/no + how), and on output zarr version.
</acceptance_criteria>
</task>
